📚 SHELLS HACK ログ分析ガイド

目次 概要・データ構造 meta - ゲーム設定 turns - ターン別イベント result - ゲーム結果 AI分析ガイド LLM分析プロンプト例
概要・データ構造

ゲームプレイの全ターンをイベントログ形式で記録。Firebase Realtime Databaseに保存。

Firebase パス構造

game_logs/{sessionId}/
├── meta                    // ゲーム開始時の設定・選択
├── turns/                  // ターン別イベントログ
│   ├── 1_blue              // ターン1 Blue側
│   ├── 1_red               // ターン1 Red側
│   ├── 2_blue
│   └── ...
├── result                  // ゲーム終了時の結果
├── createdAt               // ログ作成日時
└── completedAt             // ゲーム完了日時

セッションID形式

S + 8桁英数字（例: S1a2b3c4d）。ソロ・オンライン共通。

meta - ゲーム設定

ゲーム開始時に記録。プレイヤーが選択した企業・攻撃者・アーキテクチャ情報。

フィールド型説明AI分析での用途
sessionIdstringセッション識別子ログ紐付け
timestampISO8601ゲーム開始日時時系列分析
hostnamestring接続元ホスト名不正アクセス検出・監査
trainingTagstring研修タグ（例: "CORP_2025Q1"）研修セッション別分析・feedback/followupとの紐付け
mode"solo" | "online"プレイモードソロ/対戦別傾向分析
turnCountnumber設定ターン数（5-30）ゲーム長さ別分析
company.namestring選択企業名難易度別分析
company.fundnumber企業の資金力（1-3）リソース制約分析
company.robustnumber企業の堅牢性（1-3）初期防御力分析
actor.namestring選択攻撃者名攻撃者タイプ別分析
actor.fundnumber攻撃者の資金力（1-3）攻撃リソース分析
actor.technumber攻撃者の技術力（1-3）攻撃成功率との相関
actor.purposestring攻撃目的目的達成パターン分析
componentsstring[]配置コンポーネント一覧アーキテクチャ傾向分析
adjacencyobjectコンポーネント間の隣接関係攻撃経路分析
💡 分析ヒント: trainingTagを研修単位または個人単位で付与することで、全体分析だけでなく研修セッション別・個人別のゲーム行動分析が可能。フィードバックフォーム（training_feedback）との紐付けにも使用。

turns - ターン別イベントログ

各ターンのBlue/Red両側の行動を個別に記録。キー形式: {turn}_{team}

Blue防御側 Red攻撃側

Blue Team（防御側）フィールド

フィールド型説明AI分析での用途
turnnumberターン番号（1-N）時系列分析
team"blue"チーム識別フィルタリング
fundsAtStartnumberターン開始時コストリソース管理分析
fundsGainednumber獲得コスト収入パターン
fundsSpentnumber消費コスト投資判断分析
eventCardDrawnstring | null引いたイベントカード名イベント影響分析
cardDrawnstring | null引いた対策カード名カード選択傾向
action.type"countermeasure"アクション種別-
action.cardstring使用カード名対策選択パターン
action.targetstring適用先コンポーネント防御優先度分析
action.costnumberコストコスパ分析
stockPriceBeforenumberアクション前株価被害追跡

💡 分析ヒント: action = null の場合はパス（行動なし）。コスト不足か戦略的判断かをfundsAtStartで判別可能。

AI分析ガイド

LLM向けログ仕様（コピペ用）

AI分析Copilotでログ投入前にこのコンテキストを貼り付けることで、LLMがログ構造を正確に理解できる。

## SHELLS HACK ゲームログ仕様

### ゲーム概要
サイバーセキュリティ攻防シミュレーション。Blue(防御)とRed(攻撃)が交互にターンを進行。
勝利条件: Red=株価50以下or攻撃目的達成、Blue=規定ターン終了時に株価50超で生存

### ログ構造
game_logs/{sessionId}/
├── meta: ゲーム設定(企業/攻撃者/コンポーネント/隣接関係)
├── turns/{N}_{blue|red}: ターン別行動ログ
├── result: 勝敗・終了条件・最終状態
├── createdAt, completedAt: タイムスタンプ

### meta主要フィールド
- mode: "solo"|"online"
- turnCount: 設定ターン数(5-30)
- company: {name, fund(1-3), robust(1-3)}
- actor: {name, fund(1-3), tech(1-3), purpose}
- components: 配置コンポーネント名配列
- adjacency: {コンポーネント名: [隣接コンポーネント名]}
- trainingTag: 研修識別タグ(nullable)

### turns[blue]主要フィールド
- fundsAtStart/fundsGained/fundsSpent: コスト管理
- eventCardDrawn/cardDrawn: ドローしたカード(nullable)
- action: {type:"countermeasure", card, target, cost} または null(パス)
- stockPriceBefore/stockPriceAfter: 株価変動
- defenseState: {コンポーネント名: {detection(0-4), defense(0-4), recovery(0-4)}}

### turns[red]主要フィールド
- action: {type:"attack", card, target, category, cost, success:boolean}
- category: "偵察"|"武器化"|"侵入"|"目的実行" (キルチェーン)
- attackState: {reconned:[], intruded:[], buffs:{カテゴリ:値}, bypassAdjacency:boolean}

### result主要フィールド
- winner: "red"|"blue"
- finalStockPrice: 0-100
- endCondition: "株価50%以下"|"目的達成"|"turns_exhausted"
- summary: {assetsCompromised:[], intrudedComponents:[]}

### キルチェーン進行
偵察(対象を偵察済みに)→武器化(バフ付与)→侵入(偵察済み隣接に侵入)→目的実行(資産窃取/株価下落)

### 防御レベル(detection/defense/recovery)
0=無防備、1=基本、2=標準、3=強化、4=最大。攻撃成功率に影響。

### 分析観点
- 敗因: Red勝利時、侵入成功ターンのdefenseState/attackStateから弱点特定
- 効果的対策: Blue勝利時のaction.cardの傾向
- 攻撃経路: intrudedの順序とadjacencyの関係
- リソース管理: fundsAtStart/fundsSpent/fundsGainedの推移

## カードマスター情報

### 攻撃カード一覧
偵察カテゴリ(4種):
- OSINT(cost:0,lv:偵察1) - 対象:全て(直接攻撃可能)
- アクティブスキャン(cost:2,lv:偵察2) - 対象:全て(直接攻撃可能)
- ソーシャルエンジニアリング(cost:4,lv:偵察3) - 対象:業務端末
- パッシブ偵察(侵入後)(cost:3,lv:偵察2) - 対象:全て(隣接必須), 条件:隣接侵入済み

武器化カテゴリ(5種):
- フィッシングサイト構築(cost:2) - 効果:侵入+1, 条件:攻撃対象が業務端末
- エクスプロイトキット入手(cost:2) - 効果:侵入+1
- RaaS契約(cost:2) - 効果:実行+1, 条件:ランサムウェア展開使用時
- ゼロデイ開発(cost:2) - 効果:侵入+2, 条件:技術力★★★以上
- マルウェア開発(cost:2) - 効果:実行+2, 条件:技術力★★以上

侵入カテゴリ(10種):
- 標的型フィッシング(cost:3,lv:侵入2) - 対象:業務端末
- 水飲み場攻撃(cost:2,lv:侵入1) - 対象:業務端末
- VPN脆弱性悪用(cost:3,lv:侵入2) - 対象:VPN Gateway
- Webアプリ脆弱性攻撃(cost:3,lv:侵入2) - 対象:Web AP
- インジェクション攻撃(cost:2,lv:侵入1) - 対象:Web AP
- IdP侵害(cost:3,lv:侵入2) - 対象:IdP
- サプライチェーン攻撃(cost:3,lv:侵入2) - 対象:業務端末,Web AP
- 内部者買収(cost:5,lv:侵入3) - 対象:業務端末
- ラテラルムーブメント(脆弱性悪用)(cost:1,lv:侵入2) - 横展開,対象:Storage,DB,Backup,運用端末or踏み台サーバ,条件:隣接侵入済み
- ラテラルムーブメント(認証情報悪用)(cost:1,lv:侵入2) - 横展開,対象:全て,条件:隣接侵入済み

目的実行カテゴリ(5種):
- データ窃取・外部送信(cost:2,lv:実行2) - 漏洩,対象:全て,条件:侵入済み
- ランサムウェア展開(cost:2,lv:実行2) - 漏洩,対象:業務端末,運用端末or踏み台サーバ,Backup,Storage,条件:侵入済み

- DDoS攻撃(cost:4,lv:実行1) - 停止,対象:Web AP,DNS,条件:なし
- インフラ破壊(cost:3,lv:実行2) - 停止,対象:Web AP,運用端末or踏み台サーバ,Storage,DB,Backup,条件:侵入済み

### 攻撃者タイプと勝利条件
- 国家支援金銭調達: 株価30%下落 or 金融資産窃取
- 国家支援産業スパイ: 企業秘密窃取
- 産業スパイ集団: 社内認証情報 or 企業秘密窃取
- ハクティビスト: サービス停止 or 株価20%下落

### データ連携（feedback併用時）
game_logsとtraining_feedbackはtrainingTagで紐付け可能。
- game_logs/{sessionId}/meta.trainingTag
- training_feedback/responses/{id}.trainingTag
- training_feedback/followup_responses/{id}.trainingTag
feedbackにはq1_industry(業種)、意識変化スコア等を含む。

主要分析テーマ

#テーマ使用データ期待される知見
1敗因分析（アンチパターン検出）result.winner="red" + turns + defenseState侵入成功時のBlue側防御状態から弱点パターンを抽出
2効果的対策分析result.winner="blue" + turns[blue].actionBlue勝利時に使用された対策カードの傾向・優先度
3攻撃経路分析turns[red].action + meta.adjacency + attackState.intruded頻出する侵入経路・狙われやすいコンポーネント
4研修フィードバック生成単一ログ全体個人向けの良かった点・改善点・次回アクション
5複数ログの傾向分析複数ログの統計研修全体の敗因パターン・対策効果ランキング

LLM分析プロンプト例

Admin画面のAI分析Copilotで使用するプロンプト例。上記「主要分析テーマ」と番号が対応。

使用手順

  1. 上記「LLM向けログ仕様」をコピーしてAIに投入
  2. ログデータを投入（業種別・成長追跡の場合はfeedbackも）
  3. 以下のプロンプト例を投入

1. 敗因分析（アンチパターン検出）

このゲームログを分析し、Blue側の敗因を特定してください。
- どのターンで致命的なミスがあったか
- 侵入成功時のdefenseStateから防御の穴を特定
- どの対策を優先すべきだったか
- 攻撃者の侵入経路と、それを防ぐ方法

2. 効果的対策分析

Blue勝利のゲームログを分析し、効果的だった対策を特定してください。
- 使用された対策カードとその適用先
- 攻撃を防いだ決定的な対策
- コスト対効果の高かった対策
- 対策の優先順位の妥当性評価

3. 攻撃経路分析

このゲームログから攻撃経路を分析してください。
- 最初に狙われたコンポーネント
- adjacency（隣接関係）に沿った侵入の流れ
- 横展開のパターン
- 狙われやすいコンポーネントの特徴

4. 研修フィードバック生成

このプレイヤーのゲームログから、研修フィードバックを作成してください。
- 良かった点（効果的だった対策・判断）
- 改善点（見落としていたリスク・優先度ミス）
- 次回意識すべき具体的なアクション

5. 複数ログの傾向分析

これらのゲームログ（N件）を分析し、傾向をまとめてください。
- 頻出する敗因パターン
- 最も狙われやすいコンポーネント
- 効果的だった対策カードのランキング
- 研修全体への改善提案

6. 業種別傾向分析（feedback連携）

ゲームログとフィードバックデータを照合し、業種別の傾向を分析してください。
（trainingTagで紐付け、feedbackのq1_industryで業種を取得）
- 業種ごとの勝率
- 業種別の典型的な弱点（狙われやすい経路）
- 業種特有の対策優先傾向
- 業種別の意識変化パターン

7. 学習効果測定（成長追跡）

同一タグの複数ゲームログを時系列で分析してください。
- 回を重ねるごとの勝率変化
- 改善された点（対策選択・優先度判断）
- 繰り返し発生している課題
- feedbackの意識変化スコアとの相関
- 次のステップとして推奨する学習項目
